Lab 13: The Game of Connect 4

Connect-4 is a two-player game. The game involves the two players alternating turns. The playing board consists of a vertical grid with columns and rows that have holes in them as illustrated in Figure below. At his/her turn, a player drops a plastic game piece, called a checker, into the top of column of his/her choice. The checker falls until it hits the bottom of the column or lands on top of a previous checker in that column. Player 1 has black checkers, and player 2 has red checkers. The game play continues until one of the players gets 4 of their checkers in a contiguous line. The line can be horizontal, vertical, or diagonal.

This assignment requires you to implement a Connect-4 game in ARM assembly language. The game will interact with the two players via the terminal and the keyboard.

1. Rules

In our variation of Connect 4, the game board consists of a 6 by 7 grid.

The rules are as follows:

  • Each player takes turns dropping a checker into a column.
  • A player cannot remove checkers from a column.
  • A player cannot insert a checker into a column that is full (that is, a column where every row is occupied by a checker).
  • The game ends when a player places four of their checkers in a contiguous line (i.e., next to one another) vertically, horizontally, or diagonally.
  • A tie occurs if every location on the grid is occupied but no player has four consecutive checkers.

2. What to do

At the beginning of the game you should display the initial state of the game board. You should use the ASCII character “|” (ASCII code 0x7C) to represent the walls of the board, ”-” (ASCII code 0x2D) to represent an empty location in the grid, “O” (ASCII code 0x4F) to represent a checker of Player 1, and “X” (ASCII code 0x58) to represent a checker of Player 2. To make your grid easier to read, output a space between each column. The initial display of the (empty) game board should appear as follows:

| - - - - - - - |
| - - - - - - - |
| - - - - - - - |
| - - - - - - - |
| - - - - - - - |
| - - - - - - - |
  1 2 3 4 5 6 7

Player 1 always goes first, and play alternates between Player 1 and Player 2. At the beginning of each turn you should output which player's turn it is, and prompt the player for her move. For Player 1 this should appear as follows: 

  Player 1, choose a column:

To specify which column to drop a checker into, the player should input a single digit from 1 to 7. The number (from 1 to 7) specifies which column is selected, from left to right. For example, if the player wants to drop a checker into the 2nd column from the left, they would type 2. Your program must make sure the player's input referred to a valid column. If the player's move is invalid, you should output an error message and prompt the same player for a move. For example, if it is Player 1's turn: 

  Player 1, choose a column: D 
  Invalid move. Try again. 
  Player 1, choose a column: 8 
  Invalid move. Try again. 
  Player 1, choose a column:

Your program should keep prompting the player until a valid move is chosen.

After a player has chosen a valid move, you should update the state of the game board to reflect the move, re-display the updated game board, and check for a winner. If there is no winner, you should continue with the next player's turn.

When a player has won or a tie has occurred, the game is over. At this point, your program should show the final board and display the outcome of the game. For example, if Player 2 has four adjacent checkers, your program should output the following: 

  Player 2 Wins.

If a tie occurs, your program should output the following: 

  Tie Game.

4. Technical Details and Constraints

In an attempt to guide you toward “good” assembly code, we have created a few implementation rules. These rules will be a significant portion of the lab grade, so follow them closely, and ask questions if anything is unclear.

4.1 Create and use your own functions

Do not repeat chunks of code. Even though some compilers will optimize code by copying small functions inline, we want you to create your own functions and use the conventional ARM calling conventions to pass arguments, similar to lab 9 (calling conventions). This will make your code simpler and easier to follow.

4.2 Use Local Variables

To help you limit the number of global variables used (or static variables), you may only store your 2-D array and string constants as global variables. All other variables you use must be local variables implemented in registers or on the stack.

Local variables can be stored on the stack by using the stack pointer (sp, r13). You can read about the stack in detail and its' uses in chapter 10 of resource 2 here. Some important registers while calling functions and declaring local variables are briefly described below.

Linking Register

The linking register (lr, r14) saves the pc before a function call.

Stack Pointer

The sp points to the top of the stack and grows down, which means successive local variables stored on the stack are stored in decreasing memory locations. Along with storing local variables, each function call should store important information about the previous state of the machine. The stack pointer must be 8 byte aligned before making a function call.

Frame Pointer

The frame poiner (fp, r11) stores the sp after a function call, but before local variables are declared. So, the memory locations between the sp and fp contain the overhead of calling a function, as well as any local variables created during the function call. The data stored in the stack frame may sometimes be accessed directly via the sp. However, as the sp is variable during the activation of the routine, memory locations within the stack frame are more typically accessed via a separate register (the fp) which makes relative addressing simpler.

For an example of how the stack works with local variables and calling functions, click HERE (2).

4.3 Put the terminal in raw mode

Putting the terminal in raw mode will allow anything you type in the terminal be directly passed to your program. For example, if you are playing the game and want to select column 2, you can simply press 2 without pressing enter. Bring the terminal back to “cooked” mode after your connect 4 program ends. In C, this is how you put the terminal in raw mode or cooked mode. You can read about raw mode HERE (3) and cooked mode HERE (4). 

system("stty raw");
system("stty cooked");

You should know how to call these functions and pass the correct arguments in assembly. Refer to lab 9 if you don't remember.

If your program aborts, it may leave the terminal in raw mode. To get back to normal, you can type: 

   stty cooked

at a shell prompt. If that doesn't work, you can try typing: 

    reset

4.4 Use read() and write(), not scanf() printf()

Since we are using the terminal in raw mode, we have to use read() instead of scanf() to avoid pressing enter every time. We will also use write() instead of printf() to avoid using formatting characters. As you know from lab12, you can use the following arguments to invoke read and write: 

char   ch;
read(0, &ch, 1);   //read from stdin, store into variable ch, and read exactly 1 byte.
write(1, &ch, 1);  //write to stdout, write variable ch, and write exactly 1 byte.

You should know how to call these functions and pass the correct arguments in assembly. Refer to lab 9 if you don't remember.

8. Clarifications (updated as question come in, check often)

  1. Any function you write must also be in ARM assembly language and must be contained in the single connect4.s file.
  2. read() and write() use ASCII values to communicate with the terminal. So if you want to enter a column number “5” from the keyboard and use that value as a column offset, you will have to convert the ASCII value “5” (0x35) to an integer 5.
  3. You cannot declare static variables in the text section. For example, you cannot have a line similar to: “variable_x: .word …” or ”.comm x, 4” in the .text section. The purpose of this is we want you to use registers and the stack as much as possible.
  4. Your global array must be a single array (we want you to index into your array. You can decide to save it as row or column major order). For example ”.comm array, 42” would declare an array with 42 bytes.
  5. Do not email the TA's your code for submission. Use the “turnin” instructions.
  6. There was a typo in the stack example. In the function “foo”, the 2nd to last and 3rd to last instructions were switched. It's corrected now.
  7. Your connect4.s assembly program should put the terminal in “raw mode”. Put the string “stty raw” in r0, then use “bl system” to do this.
